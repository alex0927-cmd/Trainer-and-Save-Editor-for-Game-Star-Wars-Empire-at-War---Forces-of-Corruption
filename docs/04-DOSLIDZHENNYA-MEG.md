# 04. Дослідження Config.meg: формат архіву та витягування XML

## Навіщо це було потрібно

У `data2\Data\XML\` спочатку було лише **7 файлів**. Для повноцінного мода потрібні сотні XML (`UNITS_*.XML`, `PLANETS.XML`, `STARBASES.XML`…). Вони знаходяться всередині:

```
data2\Data\Config.meg  (~18 367 559 байт)
```

---

## Спроба 1: стандартний MEG v1/v2 (Petroglyph)

Документація [GlyphX Tools / Petrolution](http://www.petrolution.net/item-197) описує класичний заголовок:

```
+0x00  flags     uint32   0xFFFFFFFF або 0x8FFFFFFF
+0x04  id        uint32
+0x08  numFiles  uint32
+0x0C  filenamesSize uint32
```

**Результат перевірки `data2\Config.meg`:**

```
Перші 16 байт (hex): 16 04 00 00 16 04 00 00 42 00 44 41 54 41 5C 53
flags = 0x416 (НЕ 0xFFFFFFFF)
```

Магічних байтів `FFFFFFFF` у файлі **немає**.

**Висновок:** ця збірка FOC використовує **нестандартний або похідний** формат індексу, а не класичний MEG header v1.

## Спроба 2: аналіз власного індексу

### Заголовок

```python
count1, count2 = struct.unpack_from("<II", data, 0)
# count1 = 1046, count2 = 1046  (однакові)
```

### Таблиця імен файлів

Починається з offset `8`:

```
repeat count раз:
    uint16 name_length
    char[name_length] path   # напр. "DATA\XML\PLANETS.XML"
```

- Кінець таблиці імен: offset **48081**
- Кількість записів: **1046**
- XML-імен у таблиці: **628**

### Таблиця зміщень (гіпотеза)

Після імен — `1046 × 16` байт = **16736** байт.  
Кінець таблиці: offset **64817**.

Кожен запис — **4 × uint32** (16 байт):

```
(e0, e1, e2, e3)
```

### Проблема: імена ≠ зміщення за індексом

При спробі зчитати файл за індексом імені `PLANETS.XML` (індекс 544):

```
entry = (435, 134837, 7034155, 928)
offset e3=928 → вміст НЕ XML планет, а фрагмент Lua
```

Але **реальний** `PLANETS.XML` знайдено в бінарнику за вмістом:

```
offset 9728451, size 334189 байт
<?xml version="1.0"?>
<Planets>
    <Planet Name="Galaxy_Core_Art_Model">
```

Цей блок відповідає запису з **іншим індексом** (690 у таблиці), де ім'я файлу — зовсім інший (`STORY_PLOTS_*.XML`).

**Критичний висновок:** у цій версії MEG **порядок імен файлів не збігається 1:1 з порядком записів offset-таблиці** (або таблиця має іншу семантику полів). Надійний екстрактор «по індексу» **не працює**.

## Спроба 3: пошук XML за вмістом (вдалий метод)

Алгоритм:

1. Знайти всі `<?xml` у бінарнику `Config.meg`.
2. Межі файлу = від одного `<?xml` до наступного.
3. Ідентифікувати за кореневим тегом (`<Planets>`, `<GameConstants>`, `<Factions>`).

Результат:

| Файл | Offset | Розмір | Як ідентифіковано |
|------|--------|--------|-------------------|
| `PLANETS.XML` | 9728451 | 334189 | `<Planets>` + `Planet Name=` |
| `GAMECONSTANTS.XML` | 18069037 | 298522 | `<GameConstants>` |
| `FACTIONS.XML` | — | 132362 | `<Factions>` + Empire+Rebel |
| `EXPANSION_FACTIONS` | 16948624 | 60547 | `Faction Name="Underworld"` |

**Всього XML-блобів:** 627

## Спроба 4: бібліотека petro-meg

```bash
pip install petro-meg  # FAILED — мережа заблокована
uv pip install petro-meg  # SUCCESS для PyInstaller, не для MEG на момент першого етапу
```

Через блокування PyPI на `pip` використано **`uv pip`**, але повноцінний MEG-екстрактор так і не був інтегрований у фінальний продукт.

## Чому не патчили Config.meg напряму

Зміна тексту всередині MEG (наприклад, `Build_Cost_Credits>500</` → `>1</`) **змінює розмір** файлу → зсуває всі наступні offset → **ламає архів**.

Безповоротне редагування MEG вимагає:

1. Повної розпаковки;
2. Редагування;
3. Перезбірки MEG (MegBuilder).

Тому обрано шлях **розпакованих XML у `Data\XML\`** — безпечний і стандартний для моддерів.

## Практичний висновок для проєкту

1. **Для модів** — класти змінені файли в `Data\XML\` (працює).
2. **Для масового rebalance юнітів** — потрібен MegEditor з petrolution.net або petro-meg.
3. **Для редактора** — працюємо з тим, що вже на диску + можливість витягнути `PLANETS.XML` окремо.
